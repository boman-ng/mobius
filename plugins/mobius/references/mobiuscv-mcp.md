# MobiusCV MCP

MobiusCV is the recorded external review entry point for Mobius. Use it only for an explicitly
targeted Mobius goal.

## Rules

- Every review is stateless.
- Every review receives a JSON packet envelope generated by `packet-create`.
- Each packet is one-shot review input; a `packet_id` may be recorded only once.
- CSV files and CSV rows are not packet transport.
- The packet is a frozen local ledger and evidence-reference index, not a full evidence archive.
- Reviewers start from `packet.ledger`, `packet.coverage`, and `packet.refs`. Refs are starting
  points for auditing claims, not exclusive evidence and not proof by themselves.
- Prior review chat is not scope for a new review.
- `delta_review` gates one stage only and checks that stage's scope, work, gate, recovery, budget,
  linked acceptance rows, proof obligations, and evidence.
- `exit_review` checks the full active required plan DAG and acceptance matrix from scratch.
- Missing, invalid, unchecked, degraded, or ambiguous reviewer output is not pass.
- Review failures return structured `ok=false` / `persisted=false`; no success-shaped fallback.
- MobiusCV has no runtime dependency on the separate CrossReview skill.
- Terminal goals with `accepted` or `blocked` verdicts cannot receive new recorded
  reviews.

## Skill/MCP/CLI/Hook Responsibility Boundary

| Surface | Owns | Must Not Own |
| --- | --- | --- |
| `mobius-plan` skill | User-facing workflow for creating, validating, and locking a complete contract. | Hidden policy decisions, CSV writes outside CLI, or alternate contract semantics. |
| `mobius-loop` skill | User-facing full-plan loop for executing locked stages, recording evidence, creating packets, and calling recorded review until a real stop gate. | Manual interpretation of reviewer prose, manual loop advancement, or final acceptance. |
| MobiusCV MCP | Prompt building, host reviewer result normalization, Kimi adapter execution/parsing, reviewer result assembly, and calling CLI persistence APIs. | Direct CSV writes, terminal verdict computation, bypassing CLI validation, or relaxing packet/contract errors. |
| `mobius.py` CLI/ledger engine | CSV state transitions, contract validation, packet validation, one-shot packet enforcement, loop state, acceptance status, and verdict derivation. | Kimi process lifecycle or reviewer prompt/tool execution. |
| Hooks | Guardrails for direct protected-ledger writes and explicit false completion claims. | State mutation, reviewer execution, goal inference from ambient `.mobius` state, loop advancement, or verdict computation. |

## Review Gate Policies

```text
delta_light: delta_review, level 1, required reviewer codex-subagent
delta_kimi: delta_review, level 2, required reviewers codex-subagent and kimi-code
exit_strict: exit_review, level 2, required reviewers codex-subagent and kimi-code
```

`mobius_cv_record_delta_review` defaults to `delta_light` for speed on ordinary stage gates.
Callers may request level 2 or pass `input_refs.review_policy={"name":"delta_kimi"}` when a stage
needs the external Kimi reviewer. `mobius_cv_record_exit_review` always uses `exit_strict`.

Pass aggregation is derived from the canonical policy recorded in `input_refs.review_policy`. A pass
requires all policy-required reviewers to complete with pass, full required-acceptance coverage, no
degraded reviewers, and no failing completed reviewer.

Recorded reviewer output is verification input for the CLI loop. Concrete revisions, missing
evidence, stale packets, and retryable reviewer failures are returned as loop actions for the
agent to repair or rerun. MobiusCV records the review result; it does not mutate `loop.csv` outside
the CLI persistence API and it does not decide final acceptance.

An exit review `fail` with checked active acceptance ids is repair input. The CLI routes the
earliest affected stage back to `running`; it does not turn ordinary repairable exit feedback into
a terminal blocked goal.

The MCP cannot directly invoke the current Codex host subagent. Use
`mobius_cv_build_subagent_prompt`, run the host subagent with that prompt, then pass the structured
result to the recorded review tool.

Recorded review tools fail fast when `codex_subagent_result` is missing. They do not run external
reviewers, write `cv.csv`, or consume the packet until the host subagent result is provided.

## Kimi Adapter

The default external CLI reviewer is `kimi-code`.

Startup health:

```text
discover kimi
run kimi --help
no prompt smoke by default
```

## Bundled MCP Runtime

The plugin manifest points `mcpServers` at `./.mcp.json`. The companion file defines the bundled
stdio server:

```json
{
  "mcpServers": {
    "mobius-cv": {
      "cwd": ".",
      "command": "/bin/bash",
      "args": ["./scripts/mobius_cv_mcp_server.sh"]
    }
  }
}
```

Codex starts this stdio server with `cwd` resolved at the installed plugin root. The launcher also
derives its own root from the script path before running Python, so the server does not depend on
the developer's source checkout.

MobiusCV requires `uv`. Set `MOBIUS_CV_UV` to an absolute `uv` executable path when `uv` is not on
`PATH`; otherwise the launcher discovers `uv` with `command -v uv`. It intentionally does not look
in a hard-coded home-directory path. By default `uv` creates the project environment under
`${PLUGIN_DATA}/uv-venv` when Codex provides `PLUGIN_DATA`; outside an installed plugin it falls
back to `${XDG_CACHE_HOME:-${HOME:-/tmp}/.cache}/mobius-cv/uv-venv`. Override
`UV_PROJECT_ENVIRONMENT` when a different writable cache location is required.

`mobius_cv_health(deep=true)` may run a prompt smoke check (`Reply hi`) explicitly. Policies
requiring Kimi run the real review command and degrade closed on auth, timeout, or invalid output.
Startup health remains diagnostic: a transient health failure does not permanently block later review
attempts in the same MCP process.

Kimi invocations run from a temporary directory, in an isolated process group, with nested MobiusCV
startup smoke disabled. MobiusCV cleans up the process group after success or timeout.
Because Kimi runs outside the project root, MobiusCV includes reviewer-local path context in the
prompt, including the absolute project root and absolute ledger root. The packet itself remains
root-relative and unchanged; the context is only for resolving local file references during review.

Review execution uses three timeout classes:

```text
first_event_timeout
activity_idle_timeout
hard_timeout
```

Timeouts become degraded reviewer rows with `verdict=unknown`; they are never pass. OAuth, token,
login, 401/403, and auth-network failures become `auth_unavailable` with `retryable=false` and are
not retried by MobiusCV.

Kimi reviewer output must be exactly one MobiusCV-owned result block:

```text
MOBIUS_CV_REVIEWER_RESULT
REVIEWER: kimi-code
REVIEW_MODE: delta_review|exit_review
VERDICT: pass|fail|unknown|blocked
CHECKED_ACCEPTANCE_IDS: <json array>
UNCHECKED_ACCEPTANCE_IDS: <json array>
BLOCKING_FINDINGS: <json array of strings>
REQUIRED_REVISIONS: <json array of strings>
EVIDENCE_CHECKED: <json array of strings>
NOTES: <short text>
END_MOBIUS_CV_REVIEWER_RESULT
```

Extra prose, wrong reviewer id, wrong review mode, missing end marker, or non-array fields are
invalid output and are recorded as degraded reviewer results.

## Tools

```text
mobius_cv_health({ deep?: boolean, include_commands?: boolean }) -> mobius.cv_health
mobius_cv_registry({}) -> mobius.cv_registry
mobius_cv_build_subagent_prompt(packet, review_mode) -> mobius.cv_subagent_prompt
mobius_cv_record_delta_review(project_root, session_id, goal_slug, target_plan_item_id, packet, level=1, ...) -> mobius.cv_recorded_result
mobius_cv_record_exit_review(project_root, session_id, goal_slug, packet, ...) -> mobius.cv_recorded_result
```

Recorded review tools validate:

- the goal is not terminal;
- nonempty local `goal_id` and packet `packet`;
- target goal identity;
- review mode;
- required acceptance ids;
- matching `goal.md` contract hash tail;
- matching compact packet coverage, refs, and `packets.csv` packet hash.
- the `packet_id` does not already appear in `cv.csv`.
- delta review target ids match all linked required acceptance ids for the selected stage.
- `codex_subagent_result` is present before any external reviewer runs.
- pass results include the canonical `input_refs.review_policy` used for aggregation.
- raw reviewer output, when present, is written to a local `raw_reviews/*.json` artifact while
  `cv.csv` stores only `raw_ref` and `raw_hash_tail`.

Review prompts require reviewers to inspect `scope_json`, `work_json`, `gate_json`,
`recovery_json`, `budget_json`, `acceptance_ids_json`, `evidence_required_json`, `verifier_json`,
and `review_focus_json` from the indexed CSV files. MobiusCV does not copy those contract fields
into `cv.csv`.

## Packet Input

Pass a JSON object:

```json
{
  "schema": "mobius.packet",
  "packet": "packet_delta_001",
  "goal": "2026-07-03-example",
  "mode": "delta",
  "scope": "P1",
  "ledger": {
    "root": ".mobius/runs/codex-session-s/2026-07-03-example",
    "hash": "9ab31cd"
  },
  "brief": {
    "objective": "Implement the selected stage",
    "non_goals": ["unrelated behavior"],
    "risks": ["scope drift"]
  },
  "coverage": {
    "A1": ["E1"]
  },
  "refs": {
    "E1": ["command_result", "pytest", "h:f31c018"]
  }
}
```

`ledger.root` and all evidence labels are root-relative. `ledger.hash` and ref hashes are short
local sanity checks; full hashes remain in the local ledgers. Do not pass CSV ledger rows as packet
input. After any revision, retry, or rerun, call `packet-create` again and pass the new packet
object.

## Recorded Result

```json
{
  "schema": "mobius.cv_recorded_result",
  "ok": true,
  "persisted": true,
  "goal_id": "goal_...",
  "packet_id": "packet_delta_001",
  "cv_id": "cv_delta_...",
  "review_mode": "delta_review",
  "gate": "awaiting_exit_review",
  "updated_files": ["cv.csv", "loop.csv", "verdict.csv"],
  "next_required_action": "create_exit_packet",
  "loop": {
    "schema": "mobius.loop",
    "agent_must_continue": true,
    "agent_must_stop": false,
    "next_required_action": "create_exit_packet",
    "next_command": "packet-create --review-mode exit_review",
    "packet_id": "",
    "review_mode": "exit_review"
  },
  "blocking_findings": [],
  "errors": []
}
```

`mobius_cv_record_delta_review` may update `cv.csv` and `loop.csv`.
`mobius_cv_record_exit_review` may update `cv.csv`, `acceptance.csv`, and `verdict.csv`.
Those writes are staged and committed together by Mobius; failed commands must not leave partial
review state.

## Review Attempts

MobiusCV records attempt lifecycle in `review_attempts.csv`. A normal successful review moves from
`started` to `recorded`. Failed validation or persistence moves to `failed`. Keyboard interruption
moves to `interrupted` when the MCP process can catch it; a lingering `started` attempt with no
`finished_at` is surfaced by `ledger-audit` as an interrupted/open attempt so the user can retry the
outstanding packet instead of seeing a silent pending state.
